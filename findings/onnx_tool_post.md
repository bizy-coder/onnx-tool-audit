# Differential testing onnx_tool against ONNX Runtime

The current approach to ONNX tool has been to manually curate bugs (sometimes by investigating what AI has done). This has already identified numerous bugs, but it is likely there are more "attack vectors" on other ops that may be less notable. I propose we use differential testing to resolve this — and to do this, we can use ORT (ONNX Runtime), which actually executes the model and produces ground-truth tensor shapes, dtypes, and sizes.

I built a harness that runs every ONNX operator (opsets 11–18) through both `onnx_tool` and `onnxruntime`, then compares the two. Whenever ORT executes a model cleanly but `onnx_tool` reports a different shape / dtype / memory count, that's a confirmed bug. Across 170+ ops it surfaced **134 distinct issues**.

The recent scorer update measures memory via `onnx.shape_inference` instead of `onnx_tool`, which closes most of the per-tensor accounting bugs (scalar volume, constant-uncounted, etc.). However, MACs and the **validity gate** (`g.valid_profile`) still come from `onnx_tool`, so the most impactful failure modes remain:

- **Wrong shapes cascade into wrong MAC counts.** `onnx_tool` computes node MACs from its own (often-wrong) shape inference. So when it reports `ReduceSum(BCHW)` as still `(1, 10, 30, 30)` instead of `(1, 1, 1, 1)`, every downstream Conv/MatMul that consumes that tensor gets a MAC count inflated by ~10,000×. The shape bug is the MAC bug.
- **`onnx_tool` rejects valid models entirely.** ~50 ops (`Acos`, `ArgMin`, `BitShift`, `IsNaN`, `LpPool`, `Mish`, `RandomNormal`, `ReduceL1`, `Det`, `Einsum`, `GlobalLpPool`, `Cosh`, …) raise `NotImplementedError: ... has no value_infer`. Any submission using one of these fails scoring even though it runs cleanly under ORT. Differential testing across opsets 11–18 also turned up **`Compress`** crashing with `ValueError: condition must be a 1-d array` on perfectly valid inputs, and **`Constant`** crashing with `AttributeError: 'ConstantNode' object has no attribute 'value'` for several attribute variants.
- **MAC overcounts from shape errors** (still in effect): `ReduceSum / ReduceMean / ReduceMax / ReduceMin / ReduceProd / ReduceL2` over-shape their output by ~10,000×; `BatchNormalization` reports `(1, 10, 30, 30)` instead of `(10,)` — a ~360× MAC overcount on any node consuming the BN output.
- **MAC undercounts** (cheaper-than-real): `Where` with a broadcast condition is reported with the smaller `X`/`Y` shape instead of the broadcast result, undercounting any downstream node by ~10×. `PRelu` / `Pow` / `LayerNormalization` have analogous broadcast-shape undercounts.

I also added a second differential pass that mirrors the new scorer's `calculate_memory()` (which uses `onnx.shape_inference`) and compares it directly to ORT-measured tensor sizes. Even that path has divergences for several multi-output ops (e.g. `BatchNormalization` registers extra `value_info` entries that ORT doesn't allocate), and `g.valid_profile` from `onnx_tool` still rejects ~hundreds of ORT-runnable test cases.

Full report with minimal repros, affected opsets, and which fields are required vs incidental: [link to structured_issue_report.md].

**Ask for the organizers:** would you consider scoring submissions by running them through `onnxruntime` directly and measuring the actual tensor sizes / op counts, instead of relying on `onnx_tool`'s static profiler? ORT executes the model end-to-end (so MAC and memory numbers come from real allocations), and its shape inference is the production engine — far more reliable than a separate static pass. Concretely: a single `sess.run()` plus a walk over `onnx.shape_inference.infer_shapes(model)` for MACs would give you both numbers from the same source, without the ~50-op blocklist, scalar/constant accounting holes, and broadcast-shape miscounts that the current scorer inherits from `onnx_tool`.
